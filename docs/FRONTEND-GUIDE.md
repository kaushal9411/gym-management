# Frontend Guide (`apps/tenant-web`, plus super-admin notes)

Next.js 15 App Router, Tailwind, shadcn-style components, Redux Toolkit + redux-persist, TanStack Query, axios.

## State rules (strict, established conventions)

- **TanStack Query owns ALL server state.** Redux holds only UI/session state. Never mirror API data into Redux.
- Redux slices (`src/store/index.ts`): `auth` (persisted; user + permissions + tokens), `navigation` + `branch` (persisted), `ui`, and **read-only mirrors** `theme` (of next-themes) and `tenant` (of server-resolved TenantProvider) — mirrors are NOT persisted on purpose (single source of truth lives elsewhere; sync bridges: `theme-store-sync.tsx`, `tenant-store-sync.tsx`).
- **Do not call mutations from mount effects via TanStack `useMutation`** — under React 19 StrictMode the observer can stay `pending` forever (bug proven & fixed in onboarding success-step). For fire-on-mount calls use a direct service call + local `useState` machine + `useRef` once-guard (see `features/onboarding/components/steps/success-step.tsx`).

## Provider stack (`src/providers/app-providers.tsx`)

Redux → PersistGate → QueryClient → ThemeProvider(+ThemeStoreSync) → TenantProvider(+TenantStoreSync) → AuthProvider → RealtimeProvider → chrome (TopProgressBar, SessionExpiryModal, Toaster).

## Multi-tenancy on the client

- `src/middleware.ts` extracts slug from Host (`{slug}.localhost:3001` / `{slug}.<domain>`), `?tenant=` override, or cookie → header `x-tenant-slug`.
- Root layout resolves tenant server-side (`features/tenant/resolve.ts` → `GET /public/tenants/resolve`) and paints branding CSS vars on `<body>` (`--primary`, `--tenant-secondary`). Unknown slug → `PLATFORM_TENANT` fallback; `/login` then renders `FindGymForm` instead of a dead login form.
- `apiClient` (`features/auth/services/api-client.ts`): attaches `X-Tenant-Slug` + Bearer, dedupes identical in-flight requests (AbortController), 401→refresh-once-and-retry, tenant-wide error redirects (suspended/expired/maintenance), dispatches request start/finish for the top progress bar.

## Routes (`src/app/`)

- `(auth)/`: login, register(status pages), forgot/reset-password, verify-email/otp, two-factor, invitation/[token], status screens. `register/` (own layout) = onboarding wizard; `onboarding-complete` = token handoff landing.
- `(portal)/` (RequireAuth + RequireSubscription + AppShell): dashboard, billing/*, profile, settings (tabs: account/password/notifications/**sessions** — active devices + login history), notifications, support, **users** (+ `new`, `[userId]` detail with roles/branches/overrides editors + effective permissions), **roles** (+ `new`, `[roleId]`), **permissions** (registry + matrix tabs), **invitations**, **gym-settings** (P12: `profile`/`business`/`branding`/`invoice` — cross-linked via `GymSettingsNav` tabs), **staff** (P13: list + `new` + `[staffId]` detail — see below), **members** (P14: list + `new` + `[memberId]` detail — see below), **memberships** (P14: minimal plan-catalog list+create page, backs the "Assign Membership Plan" dropdown on the member detail page).
- Also under `(portal)/`: 10 "coming soon" placeholder pages (branches, attendance, workout-plans, diet-plans, payments, income, expenses, reports — `members`/`memberships` now have real pages, superseding their placeholders) — each just renders the shared `components/coming-soon.tsx` with a module-specific icon/title, so a plan-gated nav item with no page yet 404s nowhere in the app.
- `(auth)/staff-activation/[token]/` (P13): public activation page, mirrors `(auth)/invitation/[token]/` but for an ALREADY-CREATED staff account (see BACKEND-GUIDE.md §Staff Management for why this is a separate flow from `/invitation`).
- Sidebar entry "Staff & Access" → `/users` (perm `users:read`, flag `staff`); the four IAM pages cross-link via `IamNav` tabs. "Gym Settings" → `/gym-settings/profile` (perm `settings:read`, no feature flag). "Staff" (P13) → `/staff` (perm `staff:view`, flag `staff`) — a DIFFERENT nav item from "Staff & Access": that one is generic account/role administration (IAM), this one is the Manager/Trainer/Receptionist employment-record module. "Members" (pre-existing nav item, permission bumped from the old speculative `members:read` to `members:view` in P14) → `/members`. "Memberships" (pre-existing nav item, `memberships:manage`) → `/memberships`, the plan catalog.

**Gym Settings patterns (P12)**:
- Image uploads share one util, `lib/image-to-data-url.ts` → `fileToDataUrl(file, maxDimension, quality?)` (canvas downscale → JPEG data-URL) — extracted from the P11 avatar-upload component, which now imports it too rather than duplicating the logic. No object storage exists; "public URL" = the data-URL itself, stored inline.
- Every branding page mutation calls `router.refresh()` afterward (`refreshPortalChrome()` in `branding/page.tsx`) — since the backend invalidates the tenant resolve cache on write, this re-renders the server-side root layout with fresh branding without a hard reload, which is how the sidebar/login logo & colors update live.
- All 4 pages (`profile`/`business`/`branding`/`invoice`) follow the same dirty-form shape: local `useState<FormState|null>`, `useEffect` seeds it once from the query, a `baseline` recomputed from the query data for the `isDirty` diff, and the shared `UnsavedChangesBar` for Save/Cancel — copy this shape for any future settings-style page instead of the avatar/profile-form pattern.

**Staff Management patterns (P13)**:
- `components/ui/data-table.tsx`'s `DataTableColumn.header` was widened from `string` to `React.ReactNode` (backward compatible — a plain string is still a valid ReactNode) so the Staff list page could put a sortable-column button and a "select all" checkbox in table headers. Reuse this if a future list page needs the same.
- `components/ui/confirm-dialog.tsx` (new, generic `ConfirmDialog`) — the first shared "are you sure?" dialog in the UI kit; every destructive/bulk staff action (suspend/deactivate/delete/bulk-*) uses it. Prefer it over a bespoke dialog for any future confirm-before-action need.
- Staff detail page (`(portal)/staff/[staffId]/page.tsx`) combines profile+employment fields into ONE form/Save button (backed by the single `PATCH /staff/:id`), but Branch assignment and Role assignment are separate mini-sections each with their own immediate-save button — because those are separate `PUT` endpoints, mirroring how the Users detail page splits Roles/Branches editors from the main profile form.
- The Staff list/detail pages reuse `AvatarUpload` and `UserStatusBadge` directly from the `iam` feature, and `UnsavedChangesBar`/`useUnsavedChangesWarning` directly from `gym-settings` — none of these were copied or reimplemented.

**Member Management patterns (P14)**:
- `lib/image-to-data-url.ts` gained a second export, `fileToRawDataUrl(file, maxBytes)` — the Document Management upload field needs to accept PDFs, and a PDF can't go through `createImageBitmap`/canvas like `fileToDataUrl` does. `document-upload.tsx` picks between the two based on `file.type.startsWith('image/')`.
- Member detail page (`(portal)/members/[memberId]/page.tsx`) follows the same "one big form + separate mini-sections" split as Staff: profile/medical/address/notes are one `PATCH /members/:id` form with `UnsavedChangesBar`; Branch and Trainer are separate immediate-save mini-sections (`PUT` endpoints); Membership assign/renew/upgrade/freeze/unfreeze are inline action buttons with no dirty-state tracking (each click is its own mutation, matching how the backend models them as distinct lifecycle transitions, not editable fields).
- `trainer-select.tsx` importing `useStaffList` from the `staff` feature (not duplicating a members-local trainer query) is the clearest cross-module reuse example in the codebase so far — copy this pattern (import the other feature's hook directly) before adding a parallel query for data another feature's module already owns.
- The Membership Plan catalog page (`/memberships`) was NOT part of the explicit Member Management page list in the prompt, but was added anyway as the minimum necessary support for "Assign Membership Plan" to have something real to assign — a judgment call, kept deliberately minimal (list + create only, no edit/delete UI yet).

**Membership Plans patterns (P15)**:
- `/memberships` (list, was the P14 minimal stub) is now a full DataTable page: search/pagination/sort (Plan/Plan Code/Price columns, same clickable-header pattern as Staff's list page)/status filter/duplicate/status-toggle/soft-delete/restore, all via `ConfirmDialog` for destructive or state-changing actions.
- `/memberships/new` and `/memberships/[planId]` both render the same `PlanFormFields` component with different wrapping (`new` = plain form + submit button; `[planId]` = `UnsavedChangesBar` dirty-tracking, same "one big form" shape as the Staff/Member detail pages) — this is the first page pair in the app to fully share one form-fields component between Create and Edit rather than duplicating the JSX, worth copying for any future Create+Edit pair.
- Member detail page (`(portal)/members/[memberId]/page.tsx`) Membership card gained Downgrade/Extend/Cancel controls alongside the existing Assign/Renew/Upgrade/Freeze ones, and the Freeze-side button was renamed **Resume** in the UI (calls the new `useResumeMembership()` hook, which hits `/members/:id/resume` — a separate hook from the old `useMemberStatusAction({action:'unfreeze'})` path, though both ultimately do the same thing server-side). Permission checks on this card were split from one flag (`members:assign-membership`) into four (`memberships:assign`/`renew`/`upgrade`/`freeze`), matching the new backend permission set — check `hasPermission('memberships:renew')` etc., not the old key, when touching this section again.
- Checkbox form fields need an explicit `id` prop even when only used for `onCheckedChange` state (not `htmlFor`-paired) — a live Playwright E2E run caught `plan-form-fields.tsx` shipping five `<Checkbox>` elements with no `id`, making them unselectable by any `#id` locator (and worse for accessibility/label association in general). Always give every `Checkbox`/`Input` an `id`, even if no visible `<Label htmlFor>` sits next to it.

## E2E tooling

Playwright lives OUTSIDE the repo in the session scratchpad (`e2e/` with `register-flow.js`, `iam-flow.js`, `gym-settings-flow.js`, `staff-flow.js`, `member-flow.js`, `membership-plans-flow.js`); Chromium already installed under `%LOCALAPPDATA%\ms-playwright`. Pattern: headless run against `http://prompt10test.localhost:3001`, capture console errors, Mailpit API for OTPs/tokens/activation links (fetch `/api/v1/messages`, match by recipient, regex the token out of the HTML body).

**Gotcha confirmed again in P14**: `next build` while `next dev` still holds the port corrupts `.next` — always `Stop-Process` the dev server (check `Get-NetTCPConnection -LocalPort 3001`) before a production build, then restart dev afterward. Also seen: `next build`'s "Collecting page data" step can spuriously fail with `Cannot find module for page: /some-unrelated-page` on Windows — a transient file-system race, not a real error; retrying the same build (no code changes) succeeds. Don't chase it as a bug if `tsc --noEmit` and `eslint` are both clean.

**Gotcha seen in P15**: a `ConfirmDialog`-driven status action (Activate/Deactivate) on a page reached IMMEDIATELY after another mutation (e.g. right after Duplicate navigates to the new plan's detail page) can flake in Playwright with "element was detached from the DOM, retrying" on the Confirm button — plausibly a React Strict Mode double-render/query-refetch race in dev, not a real bug (the exact same action worked fine seconds earlier on a page that had been settled longer). Fix in the test, not the app: add a short `waitForTimeout` (300-500ms) after the button that opens the dialog appears, before clicking it, and again before clicking Confirm.
